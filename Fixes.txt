  Major Flaws Found

  1. CRITICAL — Phase 3 runs on the host machine with a 1-row fake dataset

  execution_submit routes through voxel_features/mcp/tools/execution_tools.py,
  which runs code via subprocess.run([sys.executable, script_path]) on the host 
  machine. The real Coe Fairbairn data is only mounted inside the analysis
  Docker container at /workspace/input. On the host, that path doesn't exist:

  $ ls /workspace/input  # → PATH DOES NOT EXIST ON HOST

  The execution tools detect this and silently fall back:
  if not
  os.path.exists("/workspace/input/amalgamated_csvs/geochemDrillhole.csv"):
      pd.DataFrame({'longitude': [117.9], 'latitude': [-27.4],
                    'li_ppm': [10.5], 'cu_ppm': [25.2]}).to_csv(...)

  So every Phase 3 code execution runs against a single fake row. Phases 1 & 2
  correctly use the Docker analysis container (via exec_run_with_timeout), but
  Phase 3 doesn't. The agent's code typically crashes at the pearsonr call
  (needs 2+ rows), but artifacts from partial execution are saved anyway. The
  downstream spatial translation and BIC evaluation are based on nothing.

  ---
  2. SIGNIFICANT — The first feature layer always gets a synthetic BIC score, 
  not a real one
  
  When the store is empty, evaluate_new_layer returns a hardcoded result without
   computing anything:
  if not existing_layers:
      store.add_layer(...)
      return {"bic_before": 0.0, "bic_after": 0.0, "bic_delta": -1.0,
  "admitted": True, ...}
  
  The current store has exactly 1 layer (copper_mineralization). Every episode
  that calls scoring_create_feature_layer removes that layer first (per
  scoring_tools.py: store.remove_layer(name)), leaving the store empty, and then
   evaluates as the "first layer" case. So all episodes evaluated so far have
  gotten bic_delta = -1.0 (synthetic), admitted = True unconditionally, and
  reward ≈ 0.001 — effectively no learning signal.

  The store never grows beyond 1 unique layer because all episodes keep
  re-discovering "copper_mineralization." You need either a mechanism to force
  layer-name diversity, or to fix the Phase 3 data issue so different hypotheses
   actually produce different results.

  ---
  3. SIGNIFICANT — Stage 1 R² threshold of 0.0001 is essentially no filter

  The masking test threshold lowered for sparse data:
  masking_test_threshold: float = 0.0001  # Lowered for sparse geological data
  At R² ≥ 0.0001, virtually any layer will pass Stage 1, including random noise
  across 320K voxels. The whole point of the two-stage system is to separate
  "does this layer have geological predictive value?" from BIC complexity
  assessment. At this threshold the gate is non-functional. The README's claim
  that Stage 1 "prevents cheat-code layers" is not achieved.
